# 简言

在大数据时代，数据的重要性不言而喻，所谓的 "埋点" 就是获取用户基本信息、行为以及跟踪产品在用户端的使用情况，并以监控数据为基础，指明产品优化的方向。埋点类型主要分为三类：数据监控、性能监控和异常监控，常见的埋点方式主要分为三种：手动埋点、可视化埋点、无埋点。

例如某件商品的点击量 / 停留时间、不同用户的网页加载时间，请求响应时间、收集客户端网页异常……

实现前端监控，第一步肯定是将我们要监控的事项（数据）给收集起来，再提交给后台，最后进行数据分析。

# 埋点类型

  1. 数据监控

    监控用户的信息和行为，可以下分为：

    • 命令式埋点：在用户行为触发位置调用事件上报函数进行行为上报，缺点是埋点和业务耦合度比较高，工作量比较大，如：

    • 声明式埋点：通过自定义指令统一完成事件上报，使得埋点和业务代码一定程度上解耦合。

      还有一些项目基于 jquery + widget 的老方案实现，那么在这些项目中的 DOM 操作是 jquery 甚至原生 DOM API 来实现，Vue 的自定义指令就无法工作，这时候就可以考虑 MutationObserver （在 DOM3 提出，提供让开发者者感知到在某一个 DOM 节点变更的能力）

      可以考虑使用 Vue自定义指令 和 IntersectionObserver ， IntersectionObserver 可以实现元素视窗观测，一般如此考虑：

      曝光使用 IntersectionObserver 观察元素是否在视窗内，并且曝光上报只上报一次，上报之后移除观察。设定每 2 秒进行一次上报。

      如何解决曝光的漏报（定时器 2 秒之内的用户退出）和多报：

      a. 漏报：保存 localStorage，下次进入之后如果有数据则上报，如果用户再不进入，对漏报的几条数据可忽略

      b. 多报：IntersectionObserver 监听曝光，上报时候移除元素的监听

      具体可见 vue 项目埋点之指令埋点（点击和曝光）。

  2. 性能监控

    监听前端性能，主要包括监听网页或者说产品在用户端的体验。

  3. 异常监控

    监听异常捕获，通常用 try……catch 进行捕获，但一些异步操作往往无法捕获。可以用 window.onerror 进行捕获；若使用 Promise 则可以用 Node.js 提供的 unhandledRejection 和 rejectionHandled 捕获异常。

    unhandledRejection ：在一轮事件循环中，当 promise 被拒绝时，会触发该事件；

    rejectionHandled ：在一轮事件循环中，当 promise 拒绝处理后，会触发该事件；

# 埋点方式

目前常见的主要有三种:

  1. 手动埋点
    也叫 代码埋点 ，也就是自己写代码，调用埋点 SDK 的函数，在需要埋点的业务逻辑功能位置调用接口，上报埋点数据.

    像友盟、百度统计等第三方数据统计服务商。

    手动埋点 因为可自主设计，灵活性强；但开发代价比较大，每一个需要埋点的地方都得加上代码，更新埋点方案时整个流程比较长。

  2. 可视化埋点
    顾名思义，通过 可视化交互 的手段。可视化埋点是指开发人员除集成采集 SDK 外，不需要额外去写埋点代码，而是由业务人员通过访问分析平台的 "圈选功能" 来 “圈” 出需要对用户行为进行捕捉的控件，并给出事件命名。圈选完毕后，这些配置会同步到各个用户的终端上，由采集 SDK 按照圈选的配置自动进行用户行为数据的采集和发送。

    例如国外的 MixPanel，国内的诸葛 IO, TalkingData 等。

    可视化埋点 只需业务同学接入，更新埋点方案的整个周期就缩短了；但自然这种方式覆盖的功能有限，不够灵活。

  3. 无埋点
    无埋点则是前端自动采集全部事件，上报埋点数据，由后端 / 数据分析师来过滤和计算出有用的数据。现在市面上主流无埋点做法有两种，一种是预先跟踪所有的渲染信息，一种是滞后跟踪的渲染信息。

    如神策等。

    好处在于因为是采集全部，所以产品迭代时不需要关注埋点逻辑；缺点是服务器🍐山大，数据传输中的网络影响
