# 简言

options 请求与同源策略和跨域相关，在发送非简单请求时会出现。简单来说有两个功能：① 获取服务器支持的 HTTP 请求 ② 检查服务器性能 (如是否安全等)

简单来说，就是在正式请求发送之前，先发一个 options 请求，浏览器查看是否安全且成功，再发送正式请求。

# 简单请求和非简单请求

下图就是在发送 POST 请求之前，浏览器还发送了一个 OPTIONS 请求：

浏览器将 CORS 请求分为两类：简单请求和非简单请求，只有同时满足两种条件才是简单请求

  • 请求方式: HEAD,GET,POST
  • 请求头信息含有自定义属性，或 Content-Type 不属于以下三个：
    • application/x-www-form-urlencoded
    • multipart/form-data
    • text/plain

# 跨域基本流程

  1. 发送简单请求时，浏览器直接发出 CORS请求 (在请求头中增加一个 Origin 字段，表明发送者的协议 + 域名 + 端口)。

    若 Origin 不在服务器许可范围，返回的 HTTP 请求内就不会包含 Access-Control-Allow-Origin ，浏览器就会抛错；反之，返回的请求就会多出几个头信息字段 ( Access-Control-Allow-Origin , Access-Control-Allow-Credentials , Access-Control-Expose-Headers )；

    Access-Control-Allow-Origin: 表示接受的域名

    Access-Control-Allow-Credentials: 该字段可选，是一个布尔值，表示是否允许发送 Cookie。(需要手动启动)

    Access-Control-Expose-Headers: 该字段可选。

  2. 发送非简单请求时，会发送预检请求 (options)，服务器检查请求中 Origin 、 Access-Control-Request-Method 和 Access-Control-Request-Headers 字段以后，确认允许跨源请求，才会发送正式请求；反之，否定了 "预检" 请求，会返回一个正常的 HTTP 回应，但是没有任何 CORS 相关的头信息字段，浏览器就会抛错。

    Access-Control-Request-Method: 列出浏览器的 CORS 请求会用到哪些 HTTP 方法；

    Access-Control-Request-Headers: 指定浏览器 CORS 请求会额外发送的头信息字段.

    服务器回应的其他 CORS 相关字段如下:

    Access-Control-Allow-Credentials: 同上，可选字段，是一个布尔值，表示是否允许发送 Cookie。

    Access-Control-Max-Age: 该字段可选，用来指定本次预检请求的有效期，单位为秒。
